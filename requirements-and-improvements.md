# 요구사항 해석 및 개선 의견

이 문서는 과제 C의 추가 제출물인 **요구사항 해석 및 개선 의견**을 정리한다.
구현 상세와 실행 방법은 [README.md](README.md), 결정의 배경은 [decisions.md](decisions.md),
전체 요구사항과 수용 기준은 [spec.md](spec.md)에서 확인할 수 있다.

## 1. 요구사항 해석 및 가정

### 1.1 동일 이벤트와 중복 요청

동일 이벤트는 `(type, refType, refId, receiverId, channel)`이 모두 같은 요청으로 정의했다.
같은 결제나 강의에 대한 알림이라도 EMAIL과 IN_APP은 전달 채널이 다르므로 별도 알림이다.

클라이언트가 `Idempotency-Key`를 전달하면 이 키를 우선 사용한다. 키를 생략하면 위 필드
조합을 각 `문자열 길이:값` 형식으로 인코딩하고 SHA-256으로 해시해 자동 생성한다. payload는
자동 키에서 제외한다. payload가 변경되어도 같은 이벤트에 대한 알림을 중복 생성하지 않는다는
해석이다. 단, 클라이언트가 명시적 키를 사용하면 같은 키에 다른 본문을 보내는 것은 키 오용으로
판단해 `422 IDEMPOTENCY_KEY_MISUSE`를 반환한다.

중복 요청은 오류가 아니라 기존 접수 결과의 replay로 해석한다. 신규와 중복 모두 `202`를
반환하며, 중복은 기존 알림 ID와 `duplicated: true`로 표시한다.

### 1.2 비동기 처리와 전달 보장

API의 책임은 발송 완료가 아니라 발송 요청을 안전하게 접수하는 데까지로 해석했다. 따라서
요청을 `PENDING`으로 저장한 뒤 실제 발송을 기다리지 않고 `202 Accepted`를 반환한다.

요청 수준에서는 DB의 `UNIQUE(idempotency_key)`로 동일 이벤트의 알림 레코드를 한 건으로
제한한다. 그러나 외부 발송과 DB 결과 기록은 하나의 원자적 트랜잭션이 아니므로 발송 수준은
at-least-once다. 외부 발송 성공 직후 프로세스가 종료되면 상태 기록이 남지 않아 stuck
recovery 이후 같은 메시지가 다시 발송될 수 있다.

### 1.3 채널별 처리

| 구분 | EMAIL | IN_APP |
| --- | --- | --- |
| 발송 | Mock sender 호출, 실패 가능 | 저장 완료를 도달로 간주하되 동일 worker 파이프라인 사용 |
| 읽음 | 현재 범위에서 지원하지 않음 | `read_at`으로 관리 |
| 읽음 API | `400 CHANNEL_NOT_SUPPORTED` | 멱등 처리 후 `200` |

IN_APP도 EMAIL과 동일한 상태 머신과 worker 파이프라인을 통과하도록 했다. 채널별 분기를
줄이고, 향후 push gateway 같은 실제 sender가 추가되어도 처리 구조를 유지하기 위해서다.

읽음 처리는 `read_at IS NULL` 조건부 UPDATE로 수행한다. 여러 기기가 동시에 요청해도 최초
한 번만 시간이 기록되며, 이후 요청도 이미 달성된 상태로 보고 성공을 반환한다.

### 1.4 실패 분류와 재시도

- 네트워크 오류나 5xx에 해당하는 일시적 실패는 backoff 후 재시도한다.
- 잘못된 수신자나 복구 불가능한 4xx 성격의 영구 실패는 즉시 `FAILED`로 전환한다.
- 예상하지 못한 예외는 일시적 장애일 가능성을 고려해 `UNKNOWN` retryable 실패로 기록한다.
- 최대 자동 시도 횟수에 도달하면 더 이상 재시도하지 않는다.

재시도 대기는 별도 `RETRY_WAIT` 상태를 만들지 않고 `PENDING + next_attempt_at`으로
표현한다. 신규 대기와 재시도 대기 모두 worker 관점에서는 “처리 가능한 시각을 기다리는
작업”이므로 상태 수를 늘리지 않는 편이 단순하다고 판단했다.

### 1.5 서버 재시작과 stuck 작업

PENDING 작업은 DB에 남으므로 애플리케이션 재시작 후 다시 처리한다. PROCESSING 상태로
기준 시간을 넘긴 작업은 worker가 중단된 것으로 보고 실패 attempt를 기록한다. 외부 발송이
이미 수행됐는지 확정할 수 없으므로 해당 시도를 소비된 시도로 본다. 남은 시도가 있으면
PENDING으로 회수하고, 최대 시도 횟수에 도달했으면 FAILED로 종료한다.

회수 이후 이전 worker가 늦게 결과를 반환할 수 있으므로 claim마다 UUID `claim_token`을
발급한다. 결과는 `(id, PROCESSING, claim_token)`이 모두 일치할 때만 기록하며, 일치하지 않는
stale 결과는 상태와 attempt 이력을 변경하지 않는다.

### 1.6 수신 가능 여부 확인

수신자 상태의 원천은 사용자 도메인에 있고, 알림 시스템은 발송 직전에 해당 상태를 확인해
정책을 집행한다고 해석했다. 접수와 발송 사이에는 재시도를 포함한 시간 차가 있으므로 접수
시점의 확인만으로 실제 발송 시점의 유효성을 보장할 수 없기 때문이다.

| 책임 | 주체 | 내용 |
| --- | --- | --- |
| 상태의 원천 | 사용자 도메인 | 탈퇴, 정지, 채널 유효성, 수신 동의 관리 |
| 발송 요청 | 이벤트 발생 시스템 | 알림을 보내야 한다는 의도 전달 |
| 최종 판단 | 알림 시스템 | 발송 직전에 최신 상태를 확인하고 발송 또는 중단 |

과제에서는 `withdrawn-*`, `ghost-*` 패턴으로 사용자 상태 조회를 대체했다. 탈퇴 사용자는
`RECIPIENT_GONE`, 미존재 사용자는 데이터 이상 가능성을 경고하고
`RECIPIENT_NOT_FOUND`로 최종 실패 처리한다.

### 1.7 수동 재시도 정책

최종 실패 수동 재시도 API는 선택 기능이라 구현하지 않았다. 구현한다면 수동 재시도는 운영자가
장애 원인 해소를 확인한 뒤 내리는 새로운 처리 지시이므로 자동 재시도 예산과 별도의 retry
cycle을 부여하는 것이 적절하다고 판단한다.

기존 attempt 이력은 보존해야 한다. 현재 `UNIQUE(notification_id, attempt_no)`가 있으므로
단순히 attempt_count를 0으로 초기화하면 기존 이력과 충돌한다. `retry_cycle`을 추가하거나
전체 이력에서 증가하는 attempt 번호와 cycle 내 시도 번호를 분리하는 방식이 필요하다.

### 1.8 알림 보관

현재는 사용자 목록과 운영 추적을 위해 SENT와 FAILED 알림을 삭제하지 않는다. 사용자 알림함에는
실제 도달이 완료된 SENT만 노출하고 FAILED는 단건 상태 조회와 운영 추적에만 사용한다. 과제 범위에서는
보관 기간을 정하지 않았지만, 실제 서비스에서는 알림의 성격과 개인정보 포함 여부에 따라
보관·아카이빙·삭제 정책을 명시해야 한다.

## 2. 개선 의견

### 2.1 외부 발송 멱등성

가장 먼저 개선할 부분은 외부 발송의 중복 가능성이다. 현재 claim token은 stale worker가 DB
상태를 덮어쓰는 것은 막지만 이미 발생한 이메일이나 push 전송을 취소하지는 못한다.

- 발송 provider가 지원하면 `notificationId` 기반 idempotency key를 전달한다.
- 지원하지 않으면 별도 delivery ledger나 수신 측 deduplication을 적용한다.
- 발송 결과 조회 API가 있다면 불확실한 실패 후 재전송 전에 결과를 확인한다.

### 2.2 관측 가능성

운영 환경에서는 다음 메트릭과 구조화 로그가 필요하다.

- PENDING, PROCESSING, FAILED 건수와 가장 오래된 PENDING의 대기 시간
- 채널별 성공률, 처리 시간, 재시도 횟수
- stuck recovery와 executor rejection 횟수
- `notificationId`, `attemptNo`, `channel`, 처리 시간, 실패 분류를 포함한 구조화 로그
- 실패율과 대기 시간 임계값에 대한 alert

### 2.3 Stuck 판정과 lease

고정 timeout 방식은 정상적으로 느린 발송을 stuck으로 오판할 수 있다. 외부 호출 timeout을
stuck threshold보다 짧게 강제하고, 장시간 작업이 필요하다면 `lease_expires_at`과 heartbeat를
도입해 살아 있는 worker가 lease를 연장하도록 개선할 수 있다.

executor 제출 실패 시에는 stuck recovery를 기다리지 않고 현재 claim token을 조건으로 즉시
PENDING으로 되돌리면 불필요한 처리 지연을 줄일 수 있다.

### 2.4 메시지 브로커 전환

현재 DB polling은 과제 규모와 실행 편의성에는 적합하지만, 트래픽과 인스턴스 수가 증가하면
빈 polling과 선점 쿼리가 DB 부하가 된다. 이 시점에는 다음 구조를 고려할 수 있다.

```text
API transaction
  → transactional outbox
  → outbox relay
  → message broker
  → consumer
  → recipient check / sender / result recording
```

브로커를 도입해도 외부 발송의 exactly-once가 자동으로 보장되는 것은 아니다. consumer와
provider 양쪽의 멱등성 정책은 계속 필요하다.

### 2.5 조회 성능과 데이터 보관

- 사용자 알림 목록이 커지면 offset pagination을 cursor/keyset pagination으로 전환한다.
- 전체 건수가 필요하지 않은 화면은 `Page` 대신 `Slice`를 사용해 COUNT 쿼리를 제거한다.
- 단건 응답의 attempt 이력은 별도 페이지 API 또는 최근 N건으로 제한한다.
- 오래된 알림과 attempt 이력을 아카이빙하거나 보관 기간 후 삭제한다.

### 2.6 사용자 생애주기와 개인정보

탈퇴 이벤트를 구독해 대기 중인 마케팅 알림을 `CANCELED`로 전환하고, 법적 보관 의무가 없는
receiverId와 payload는 삭제하거나 비식별화해야 한다. 반면 결제·환불 같은 거래성 알림은
탈퇴 여부와 무관하게 전달 또는 보관해야 할 수 있다. 따라서 수신 가능 정책은 사용자 상태뿐
아니라 알림 타입과 수신 동의까지 입력으로 받아야 한다.

실제 API에는 인증·인가를 적용해 URL의 userId가 인증 사용자와 일치하는지 확인하고, 운영자
조회 권한과 사용자 조회 권한을 분리해야 한다. payload와 로그의 개인정보 마스킹도 필요하다.

### 2.7 스키마 마이그레이션과 배포

현재 Flyway V1은 fresh clone과 빈 DB 제출 환경을 목표로 한다. 운영 전환 시에는 원본 스키마를
V1, 변경 사항을 V2 이후로 분리하고 기존 데이터가 존재하는 상태의 migration 테스트를 추가해야
한다. MySQL 이미지는 검증한 정확한 버전으로 고정하고, rolling deployment 동안 이전·신규
애플리케이션이 같은 스키마에서 함께 동작할 수 있는지도 확인해야 한다.

### 2.8 재시도 정책 개선

현재 설정된 backoff 목록은 단순하고 테스트하기 쉽지만 대규모 장애 시 같은 시점에 재시도가
집중될 수 있다. 처리량이 커지면 exponential backoff와 random jitter를 적용하고, provider별
rate limit과 circuit breaker를 추가하는 것이 좋다.

## 3. 구현 범위에 대한 의견

필수 요구사항의 비동기 처리, 멱등성, 재시도, 다중 worker, 재시작 내구성과 stuck recovery를
우선 구현했다. 선택 기능 중에서는 기존 데이터 모델과 Querydsl 요구사항을 활용하면서 정책
범위가 명확한 사용자 목록과 읽음 처리를 구현했다.

발송 예약, 템플릿 관리, 최종 실패 수동 재시도는 각각 시간 정책, 버전 정책, retry cycle 정책을
추가로 요구한다. 제한된 과제 시간에는 기능 수를 늘리기보다 핵심 발송 흐름의 동시성과 장애
시나리오를 실제 MySQL 통합 테스트로 검증하는 편이 평가 목적에 더 부합한다고 판단했다.
