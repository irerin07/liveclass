## D-1. 중복 요청은 오류가 아니라 멱등 성공 응답으로 — 그리고 그 결정이 강제하는 구현 연쇄

### 문제

동일한 알림 발송 요청이 2번 들어온다 (호출자의 타임아웃 재시도, 이벤트 중복 발행,
동시 요청). 두 번째 요청에 무엇을 응답할 것인가? 그리고 그 응답을 만들기 위해
내부적으로 무엇이 필요한가?

### 결정

- 중복 요청에는 오류(409)가 아니라 신규와 **동일하게** `202 Accepted + 기존 알림 ID
  + duplicated: true` 를 응답한다 (Stripe·IETF Idempotency-Key 방식의 replay —
  신규/중복이 상태 코드로 갈리지 않고 본문 플래그로만 구분).
- 이를 위해 UNIQUE 제약 위반을 받은 뒤 **새 트랜잭션에서 멱등성 키로 재조회**하여
  기존(승자) 행의 ID를 알아낸다.
- 등록 INSERT는 `REQUIRES_NEW`(또는 독립 트랜잭션)로 격리하고, 예외는 트랜잭션
  경계 **밖**에서 잡는다.
- (당초 중복 = `200 OK`로 잡았으나 → `202`로 정정. 신규가 202인데 중복이 200이면
  응답 멱등성이 깨지고, 200="완료됨"이 PENDING 상태와 모순되기 때문.)

### 논리 사슬 — 하나의 설계 결정이 구현 디테일을 연쇄적으로 결정한다

각 단계는 앞 단계의 필연적 결과다. 역방향으로 "왜?"를 물으면 이 사슬을 거슬러
올라간다.

**1. 우리 API는 비동기다 → 알림 ID는 "교환권"이다.**

등록 응답은 202 — "접수했고, 결과는 아직 없다"이다. 발송은 몇 초~몇십 분 뒤(재시도
포함) 워커가 수행한다. 결과가 나중에 생기는 구조에서 호출자가 결과를 찾아올 연결
고리가 알림 ID다. 필수 요구사항인 상태 조회 API(FR-2, `GET /api/notifications/{id}`)의
주소가 ID이고, 호출자가 ID를 얻는 유일한 창구가 등록 응답이다. **FR-1의 응답이
FR-2의 입장권이다.** 여기서 ID를 안 주면 상태 조회 API는 아무도 호출할 수 없다.

**2. 멱등의 정의: 같은 요청 → 같은 결과 → 중복 요청도 같은 ID를 받아야 한다.**

중복 요청의 대표 원인은 호출자의 타임아웃 재시도다: 첫 요청의 응답을 못 받아
"등록이 됐는지 안 됐는지 모르는" 상태에서 다시 보낸다. 이때 오류를 돌려주면
호출자는 여전히 등록 여부를 알 수 없고, 결국 별도 확인 로직을 짜야 한다. 그
확인(멱등성 키로 알림 찾기)은 없어지는 일이 아니라 **서버 한 곳이 안 하면 호출자
N곳이 각자 구현해야 하는 일**이다. 호출자는 여럿(수강 신청, 결제, 스케줄러...)이고
우리는 한 곳이다. 또한 제약 위반은 비즈니스 관점에서 실패가 아니다 — "네가 원하는
결과가 이미 존재한다"는 성공 신호다. 존재하는 결과에 오류를 응답하는 것은 의미상
거짓이다.

**3. 제약 위반 예외에는 승자의 ID가 없다 → 재조회가 필요하다.**

동시 중복(레이스)에서 진 쪽 요청은 사전 조회에서 아무것도 못 봤고(그래서 INSERT로
진행), 받은 것은 duplicate key 예외뿐이다. 예외에는 기존 행의 PK가 담겨 있지 않다.
응답 계약(202 + 기존 ID)을 완성하려면 멱등성 키로 재조회해서 승자 행을 찾아야
한다. 재조회는 오류를 수습하는 절차가 아니라 **"이미 성공해 있다"는 정답을
완성하는 절차**다.

**4. 재조회는 제약 위반이 난 트랜잭션에서 할 수 없다 (rollback-only).**

제약 위반이 발생한 순간 Spring/JPA는 그 트랜잭션을 rollback-only로 마킹한다 —
커밋될 수 없는 트랜잭션이다. 예외를 catch해도 자바 코드 흐름이 살아날 뿐
트랜잭션은 살아나지 않는다. 그 트랜잭션에서 재조회를 이어가면 메서드 종료 시
커밋 시도에서 `UnexpectedRollbackException`이 터진다. 실패한 INSERT를 겪은
영속성 컨텍스트도 내부 상태가 어긋나 있어 후속 쿼리의 동작을 보장할 수 없다.

**5. 살아 있어도 승자 행이 안 보인다 (MVCC 스냅샷) → 새 트랜잭션이어야 한다.**

MySQL(InnoDB) 기본 격리 수준 REPEATABLE READ에서 일반 SELECT는 트랜잭션 시작
시점의 스냅샷을 본다. 승자 행은 진 쪽 트랜잭션이 시작된 **이후에** 커밋됐으므로
같은 트랜잭션의 재조회에는 보이지 않는다. 반면 유니크 제약 검사는 스냅샷이 아니라
인덱스의 현재 상태로 수행된다. 그래서 "방금 이 행 때문에 튕겼는데 조회하면 없는"
모순이 실제로 발생한다.

```
t1  요청 2 트랜잭션 시작, 사전 조회 → 스냅샷 확정 (이 시점의 세계)
t2  요청 1이 id=1 INSERT 후 커밋              ← 스냅샷 이후의 사건
t3  요청 2 INSERT → 제약 위반                 ← 제약 검사는 "현재"를 봄
t4  같은 트랜잭션에서 재조회 → 빈 결과(!)      ← SELECT는 t1의 세계를 봄
```

새 트랜잭션은 스냅샷을 지금 시점으로 새로 잡으므로 t2에 커밋된 승자 행이 보인다.

**요약: 상태 조회 API가 ID로 주소를 잡는다(1) → 등록 응답이 ID 발급 창구이고
중복에도 같은 ID를 준다(2) → 예외에는 ID가 없으니 재조회한다(3) → 원래
트랜잭션은 죽었고(4) 낡은 스냅샷이라(5) 새 트랜잭션에서 재조회한다.**

### 도착 타이밍별 동작 시나리오

| 케이스 | 상황 | 걸리는 방어선 | 결과 |
| --- | --- | --- | --- |
| 순차 도착 | 요청 1 커밋 후 요청 2 도착 | 1차 (사전 조회) | INSERT 없이 202 + 기존 ID |
| 완전 동시 | 둘 다 사전 조회 통과 (예: 더블클릭·이중제출) | 2차 (UNIQUE 제약) → 재조회 | 진 쪽이 202 + 기존 ID |
| 겹친 도착 | 요청 1 커밋 전에 요청 2 INSERT | 유니크 인덱스 락 대기 → 제약 위반 → 재조회 | 진 쪽이 202 + 기존 ID (요청 1 롤백 시엔 요청 2가 정당하게 성공) |

세 경로 모두 불변식은 동일하다: **DB에 알림 1건, 발송 1건, 모든 호출자는 성공
응답(202) + 같은 ID.** 더블클릭·이중제출은 "완전 동시" 케이스이며 UNIQUE 제약이
방어한다 — T2.5(같은 키 N-스레드 동시)가 이 시나리오를 검증한다.

### 기각한 대안

| 대안 | 기각 이유 |
| --- | --- |
| 중복 시 409 응답으로 종료 | ① **식별 불가능성**: 서버가 보는 건 "같은 키의 두 번째 요청" 하나뿐인데 원인은 여럿(타임아웃 재시도·버스 재전달·더블클릭)이고 와이어에서 구분 불가. 409는 오류가 아닌 정상 재시도까지 싸잡아 오분류. ② 재시도가 필요로 하는 ID를 withhold → 키로 조회하는 별도 API가 또 필요. ③ 이미 완료된 작업에 "고쳐서 다시 하라"는 4xx는 의미상 틀림. (단 **같은 키 + 다른 본문**은 재생이 아니라 모순 → 422가 정당) |
| 신규 202 / 중복 200 (당초 안) | 같은 요청에 상태 코드가 갈려 응답 멱등성 위반. 게다가 200="완료됨"이 PENDING 상태와 모순 |
| ID 없이 202만 응답 | 첫 요청(202+ID)과 응답 형태가 달라져 멱등 계약 위반 — 호출자가 첫 시도/재시도를 구분 처리해야 함. 상태 조회 API 사용 불가 |
| 멱등성 키 기반 상태 조회 엔드포인트 추가 (`GET /notifications?idempotencyKey=...`)로 ID 반환 회피 | 재조회가 별도 엔드포인트로 이동할 뿐 일이 없어지지 않음. API 표면적만 증가 |
| 사전 조회 없이 INSERT-먼저 (2차 방어만) | 동작은 하지만 일반 경로(순차 중복)에서도 매번 예외 비용 발생. 사전 조회는 흔한 경로의 최적화로 유지 |

**"중복 = 정상 트래픽"의 의미**: at-least-once + 재시도 전제 시스템에서 중복은
분산 시스템의 lost-ack 불가능성·버스 재전달·계층별 재시도로 **구조적으로 보장**되는
정상 트래픽이다(아무도 잘못하지 않아도 발생). 그래서 dedup은 엣지 케이스 핸들러가
아니라 요청 경로의 상시 부품이고, 중복 응답은 에러 분기가 아니라 성공 경로(202)다.
동기적 "유일 리소스 생성" 엔드포인트(예: 이메일 중복 → 409)와 달리, 본 엔드포인트는
비동기 수집(ingestion)이라 409-on-duplicate가 맞지 않는다.
