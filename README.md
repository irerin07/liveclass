# 과제 A — 수강 신청 시스템

> **유형:** CRUD + 비즈니스 규칙형

## 1. 배경 시나리오

* 크리에이터인 강사는 강의를 개설합니다.
* 강사는 강의의 수강 정원, 가격, 수강 기간을 설정합니다.
* 클래스메이트인 수강생은 원하는 강의에 수강 신청을 합니다.
* 강의 정원이 초과되면 수강 신청이 불가능합니다.
* 신청 후 결제가 완료되어야 수강이 확정됩니다.
* 수강 확정 후 일정 기간 내에는 취소할 수 있으며, 해당 기간이 지나면 취소할 수 없습니다.

---

## 2. 구현 범위

### 2.1 필수 구현

#### 2.1.1 강의 관리

강의 등록 시 다음 정보를 입력합니다.

* 제목
* 설명
* 가격
* 정원

  * 최대 수강 인원
* 수강 기간

  * 시작일
  * 종료일

강의 상태는 다음과 같습니다.

```text
DRAFT → OPEN → CLOSED
```

| 상태       | 설명    | 수강 신청 가능 여부 |
| -------- | ----- | ----------- |
| `DRAFT`  | 강의 초안 | 신청 불가       |
| `OPEN`   | 모집 중  | 신청 가능       |
| `CLOSED` | 모집 마감 | 신청 불가       |

구현해야 하는 기능은 다음과 같습니다.

* 강의 등록
* 강의 목록 조회

  * 상태 필터 지원
* 강의 상세 조회

  * 현재 신청 인원 포함

---

#### 2.1.2 수강 신청 관리

수강 신청 상태는 다음과 같습니다.

```text
PENDING → CONFIRMED → CANCELLED
```

| 상태          | 설명           |
| ----------- | ------------ |
| `PENDING`   | 신청 완료, 결제 대기 |
| `CONFIRMED` | 결제 완료, 수강 확정 |
| `CANCELLED` | 수강 신청 취소     |

구현해야 하는 기능은 다음과 같습니다.

* 수강 신청

  * 사용자가 특정 강의에 신청
* 결제 확정 처리

  * 외부 결제 시스템 연동은 불필요
  * 단순 상태 변경으로 대체
* 수강 취소
* 내 수강 신청 목록 조회

---

#### 2.1.3 정원 관리 규칙

* 강의별 최대 정원을 초과한 신청은 거부해야 합니다.
* 여러 사용자가 동시에 마지막 남은 자리에 신청하는 상황을 고려해야 합니다.

---

### 2.2 선택 구현

다음 기능을 구현하면 추가 점수를 받을 수 있습니다.

* 수강 취소 가능 기간 제한

  * 예: 결제 완료 후 7일 이내
* 대기열 기능

  * Waitlist
* 강의별 수강생 목록 조회

  * 크리에이터 전용
* 수강 신청 내역 페이지네이션

# 과제 B — 크리에이터 정산 API

> **유형:** 데이터 처리 / 정산형

## 1. 배경 시나리오

* 크리에이터는 강의를 개설하고 수강생에게 판매합니다.
* 플랫폼은 판매 금액에서 수수료를 제외한 금액을 크리에이터에게 정산합니다.
* 크리에이터는 자신의 정산 내역을 월별로 조회할 수 있습니다.
* 플랫폼 운영자는 특정 기간의 전체 정산 현황을 집계할 수 있습니다.
* 수강 취소가 발생하면 이미 계산된 정산 금액에서 환불 금액이 차감됩니다.

---

## 2. 구현 범위

### 2.1 필수 구현

#### 2.1.1 판매 내역 관리

판매 내역 등록 시 다음 정보를 저장합니다.

* 강의 ID
* 수강생 ID
* 결제 금액
* 결제 일시

취소 내역 등록 시 다음 정보를 저장합니다.

* 원본 판매 내역
* 환불 금액
* 취소 일시

구현해야 하는 기능은 다음과 같습니다.

* 판매 내역 등록
* 취소 내역 등록
* 판매 내역 목록 조회

  * 크리에이터별 조회
  * 기간 필터 지원

---

#### 2.1.2 정산 금액 계산 API

크리에이터별 월간 정산 정보를 조회합니다.

##### 요청 정보

* 크리에이터 ID
* 조회 연월

  * 예: `2025-03`

##### 응답 정보

* 해당 월 총 판매 금액
* 취소·환불 금액
* 순 판매 금액
* 플랫폼 수수료
* 정산 예정 금액
* 판매 건수
* 취소 건수

##### 계산식

```text
순 판매 금액 = 총 판매 금액 - 환불 금액

플랫폼 수수료 = 순 판매 금액 × 20%

정산 예정 금액 = 순 판매 금액 - 플랫폼 수수료
```

---

#### 2.1.3 정산 기간 기준

정산 기간 기준을 명확하게 정의해야 합니다.

* 판매 금액

  * 결제 완료 일시 기준
* 취소 금액

  * 취소 일시 기준
* 시간대

  * KST 기준
* 월간 조회 범위

  * 해당 월 1일 `00:00:00`
  * 다음 달 1일 `00:00:00` 미만

요구사항에는 다음과 같이 표현되어 있습니다.

```text
해당 월 1일 00:00:00 ~ 말일 23:59:59
```

---

#### 2.1.4 정산 내역 집계 API

플랫폼 운영자가 특정 기간의 전체 정산 현황을 조회합니다.

##### 요청 정보

* 시작일
* 종료일

##### 응답 정보

* 크리에이터별 정산 예정 금액 목록
* 전체 정산 예정 금액 합계

---

### 2.2 선택 구현

다음 기능을 구현하면 추가 점수를 받을 수 있습니다.

* 정산 확정 상태 관리

```text
PENDING → CONFIRMED → PAID
```

* 동일 기간 중복 정산 방지
* 정산 내역 엑셀 다운로드

  * 또는 CSV 응답
* 수수료율 변경 이력 관리

  * 과거 정산에는 당시 적용된 수수료율 사용

---

## 3. 과제별 제약 사항

* 수수료율은 우선 **고정값 20%**로 구현합니다.
* 향후 수수료율 변경 가능성을 설계에 반영하면 가산점을 받을 수 있습니다.
* 실제 결제 시스템 연동은 필요하지 않습니다.
* 판매 데이터는 직접 삽입하거나 API를 통해 등록할 수 있습니다.

## 5. 샘플 데이터 검증 시나리오

| 시나리오                      | 관련 데이터                                                         | 확인 포인트                                                                     |
| ------------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `creator-1`의 `2025-03` 정산 | `sale-1`, `sale-2`, `sale-3`, `sale-4`, `cancel-1`, `cancel-2` | 총 판매 260,000원 / 환불 110,000원 / 순 판매 150,000원 / 수수료 30,000원 / 정산 예정 120,000원 |
| 부분 환불 처리                  | `sale-4`, `cancel-2`                                           | 환불 금액 30,000원이 원결제 금액 80,000원보다 작은 경우 순 판매 금액에 정상 반영                       |
| 월 경계 취소                   | `sale-5`, `cancel-3`                                           | `sale-5`는 1월 판매로 반영하고, `cancel-3`은 2월 환불로 반영                               |
| 빈 월 조회                    | `creator-3`, `2025-03`                                         | 판매 내역이 없는 월 조회 시 0원 응답 또는 정의한 처리 방침을 일관되게 적용                               |

## 6. 추가 데이터 가이드

제공된 샘플 데이터 외에도 중요하다고 판단한 시나리오를 추가로 등록하고 테스트해야 합니다.

예시는 다음과 같습니다.

* 동일 판매 건에 대한 다수의 부분 취소
* 환불 금액 합계가 원결제 금액을 초과하는 요청
* 미래 날짜를 대상으로 한 정산 조회
* 잘못된 연월 형식
* 존재하지 않는 크리에이터 조회
* 조회 기간의 시작일이 종료일보다 늦은 경우
* 월 경계 시각의 결제 또는 취소
* 판매와 취소가 서로 다른 월에 발생한 경우
* 판매 내역이 없는 크리에이터
* 금액이 0원 또는 음수인 판매·환불 요청

# 과제 C — 알림 발송 시스템

> **유형:** 이벤트 / 비동기 / 운영 고려형

## 1. 배경 시나리오

다음과 같은 이벤트가 발생하면 사용자에게 알림을 발송해야 합니다.

* 수강 신청 완료
* 결제 확정
* 강의 시작 D-1
* 수강 취소 처리
* 기타 사용자에게 전달해야 하는 비즈니스 이벤트

알림 발송 채널은 다음과 같습니다.

* 이메일
* 인앱 알림

알림 처리 실패가 원래 비즈니스 트랜잭션에 영향을 주어서는 안 됩니다.

다만 예외를 단순히 무시하는 방식으로 이를 구현해서는 안 됩니다.

다음과 같은 일시적 장애에도 대응할 수 있어야 합니다.

* 네트워크 장애
* 외부 이메일 서버 오류
* 일시적인 타임아웃
* 서버 프로세스 중단
* 애플리케이션 재시작

또한 동일 이벤트에 대한 알림이 중복 발송되어서는 안 됩니다.

## 2. 구현 범위

### 2.1 필수 구현

#### 2.1.1 알림 발송 요청 API

알림 발송 요청을 등록합니다.

##### 요청 정보

* 수신자 ID
* 알림 타입
* 참조 데이터

  * 이벤트 ID
  * 강의 ID
  * 기타 비즈니스 식별자
* 발송 채널

지원 채널은 다음과 같습니다.

```text
EMAIL
IN_APP
```

##### 응답

API 응답은 실제 발송 완료가 아니라 **발송 요청 접수 완료**를 의미해야 합니다.

```text
요청 접수 완료
```

알림 발송은 API 요청 스레드에서 즉시 수행하지 않습니다.

---

#### 2.1.2 알림 상태 조회

특정 알림 요청의 현재 처리 상태를 조회할 수 있어야 합니다.

조회 결과에는 필요에 따라 다음 정보가 포함될 수 있습니다.

* 알림 요청 ID
* 현재 상태
* 재시도 횟수
* 마지막 실패 사유
* 생성 시각
* 처리 시작 시각
* 발송 완료 시각
* 다음 재시도 예정 시각

---

#### 2.1.3 사용자 알림 목록 조회

수신자 기준으로 알림 목록을 조회합니다.

다음 필터를 지원해야 합니다.

* 전체
* 읽음
* 읽지 않음

---

#### 2.1.4 알림 처리 상태 관리

알림 처리 상태를 적절히 정의하고 상태 전이 조건을 설계해야 합니다.

다음 내용을 포함해야 합니다.

* 초기 접수 상태
* 처리 대기 상태
* 처리 중 상태
* 발송 성공 상태
* 재시도 대기 상태
* 최종 실패 상태

발송 실패 시 다음 정책을 설계해야 합니다.

* 재시도 가능한 오류와 재시도 불가능한 오류 구분
* 최대 재시도 횟수
* 재시도 간격
* Backoff 정책
* 최종 실패 처리
* 실패 사유 기록

---

#### 2.1.5 중복 발송 방지

동일한 이벤트에 대해 알림이 중복 발송되어서는 안 됩니다.

다음 상황을 모두 고려해야 합니다.

* 동일한 요청이 순차적으로 여러 번 들어오는 경우
* 동일한 요청이 동시에 여러 번 들어오는 경우
* 서버 재시작 후 같은 알림이 다시 처리되는 경우
* 처리 결과 저장 전에 워커가 중단되는 경우
* 다중 인스턴스가 같은 알림을 동시에 조회하는 경우

---

#### 2.1.6 비동기 처리 구조

알림 발송은 API 요청 스레드와 분리되어 처리되어야 합니다.

실제 메시지 브로커는 설치하지 않아도 됩니다.

다만 향후 다음과 같은 메시지 브로커 기반 구조로 전환할 수 있도록 설계해야 합니다.

* Kafka
* RabbitMQ
* Amazon SQS
* 기타 메시지 큐

애플리케이션 내부 구현을 하더라도 다음 역할이 분리되는 구조가 권장됩니다.

```text
알림 요청 접수
    ↓
알림 요청 저장
    ↓
비동기 처리 대상 조회
    ↓
알림 발송
    ↓
결과 및 상태 저장
```

---

#### 2.1.7 운영 시나리오 대응

다음 운영 상황에 대한 대응 방법을 설계해야 합니다.

##### 처리 중 상태 장기 지속

알림이 처리 중 상태로 변경된 후 일정 시간 이상 완료되지 않는 경우 복구할 수 있어야 합니다.

예시는 다음과 같습니다.

* 처리 제한 시간 초과 여부 확인
* 처리 중 상태를 재시도 대기 상태로 복구
* 워커 소유권 또는 처리 시작 시각을 이용한 복구
* 스케줄러를 통한 장기 처리 건 탐지

##### 서버 재시작

서버가 재시작되더라도 처리되지 않은 알림이 유실되어서는 안 됩니다.

다음 알림을 재처리할 수 있어야 합니다.

* 접수 완료 후 아직 처리되지 않은 알림
* 재시도 예정 알림
* 비정상 종료로 처리 중 상태에 남아 있는 알림

##### 다중 인스턴스

여러 애플리케이션 인스턴스가 동시에 실행되더라도 동일 알림을 중복 처리해서는 안 됩니다.

다음과 같은 방법을 고려할 수 있습니다.

* DB Lock
* 낙관적 락
* 조건부 상태 변경
* `SELECT ... FOR UPDATE SKIP LOCKED`
* 처리 소유권과 만료 시간
* 고유 제약조건
* 분산 락

구체적인 구현 방식과 선택 이유를 설명해야 합니다.

### 2.2 선택 구현

다음 기능을 구현하면 추가 점수를 받을 수 있습니다.

#### 2.2.1 예약 발송

특정 시각에 알림이 발송되도록 예약할 수 있습니다.

예시는 다음과 같습니다.

* 강의 시작 하루 전
* 결제 기한 만료 전
* 수강 종료 후 후기 요청

---

#### 2.2.2 알림 템플릿 관리

알림 타입별 메시지 템플릿을 관리합니다.

예시는 다음과 같습니다.

* 수강 신청 완료 템플릿
* 결제 완료 템플릿
* 수강 취소 템플릿
* 강의 시작 예정 템플릿

채널별로 서로 다른 템플릿을 사용할 수 있습니다.

```text
EMAIL
IN_APP
```

---

#### 2.2.3 읽음 처리

인앱 알림의 읽음 상태를 처리합니다.

여러 기기에서 동시에 읽음 처리 요청이 발생하는 상황을 고려해야 합니다.

다음 사항을 정의할 수 있습니다.

* 읽음 처리를 멱등하게 처리하는 방법
* 이미 읽은 알림에 대한 재요청 처리
* 읽음 시각 갱신 정책
* 낙관적 락 적용 여부

---

#### 2.2.4 최종 실패 알림 보관 및 수동 재시도

최대 재시도 횟수를 초과한 알림을 별도로 조회하고 수동으로 재시도할 수 있습니다.

수동 재시도 시 다음 정책을 정의해야 합니다.

* 기존 재시도 횟수 유지
* 재시도 횟수 초기화
* 별도의 수동 재시도 횟수 관리
* 수동 재시도 실행자 기록
* 수동 재시도 사유 기록

## 3. 과제별 제약 사항

* 실제 이메일 발송은 필요하지 않습니다.

  * Mock 구현
  * 로그 출력
* 실제 메시지 브로커 설치는 필요하지 않습니다.
* 단, 실제 운영 환경에서 메시지 브로커 기반 구조로 전환 가능한 설계여야 합니다.
